# Remote Emby Version Testing Runbook

本文是 Emby Migrator 大版本兼容测试手册。凡是改动导出/导入核心逻辑、匹配策略、元数据 payload、媒体图片上传、人物信息或人物头像，都必须按本文执行远端实测。

不要在本文或测试报告中记录具体服务器地址、SSH 用户名、私钥路径、API Key、token。敏感连接信息只保存在本地私密上下文。

## 1. 适用场景

必须执行远端多版本实测的改动：

- Emby 版本兼容 profile。
- 导出包结构、manifest schema、`info.json`、`raw.json`。
- 媒体项目匹配策略。
- 元数据回写 payload。
- 媒体图片下载或上传。
- 人物信息、人物头像导出或导入。
- 并发、重试、超时、断点续跑等会影响大库稳定性的流程。

只改纯文档、纯样式或不影响导出/导入行为的小 UI 文案时，可以不跑远端实测，但要在交付说明中写明未涉及核心迁移逻辑。

## 2. 测试版本矩阵

当前至少覆盖：

- Emby 4.8.11 系列。
- Emby 4.9.5 系列。

重点媒体库：

- `日韩剧集`
- `日韩电影`

后续新增媒体库应沿用同一媒体地址/路径约定。测试匹配逻辑时不能依赖媒体库内部 ID，也不能把导入逻辑强绑定到媒体库名称。

## 3. 测试前检查

每次远端测试前先确认：

1. 当前代码已通过本地基础检查。
2. 当前 Docker 回滚基线已记录在 `docs/PROJECT_BLUEPRINT.md`。
3. 远端测试环境可连接。
4. 远端 Emby 源/目标版本号已记录在测试笔记中。
5. 测试使用的是明确的镜像标签，优先使用 `sha-xxxxxxx`，不要只写 `latest`。
6. 如果需要访问 Telegram 或外网资源，远端环境可使用 mihomo 代理端口 `7890`。

本地基础检查：

```bash
go test ./...
node --check web/assets/app.js
go build ./cmd/server
```

## 4. 远端部署约定

推荐测试命令结构：

```bash
docker run -d \
  --name emby-migrator \
  --restart unless-stopped \
  --network host \
  -e TZ=Asia/Shanghai \
  -e EMBY_MIGRATOR_PASSWORD=password \
  -v /opt/emby-migrator/data:/data \
  -v /opt/emby-migrator/config:/config \
  czppwa/emby-migrator:sha-xxxxxxx
```

如果要保留现有稳定容器，测试容器应使用不同名称、不同数据目录或先完整备份 `/opt/emby-migrator/data`。

## 5. 标准测试流程

### 5.1 导出测试

1. 登录 Web 页面。
2. 连接源 Emby。
3. 读取媒体库。
4. 选择 `日韩剧集` 和 `日韩电影`。
5. 默认全选媒体图片类型。
6. 启用人物头像。
7. 选择合理并发，例如 4、6、8 分别抽测。
8. 启动导出。
9. 检查日志是否持续输出，不应长时间无进度。
10. 下载完整日志和导出报告。

导出通过条件：

- 任务状态为完成。
- 导出总结包含媒体库数、项目数、媒体图片数、人物数、人物头像数、错误数、用时。
- 导出包目录名可区分时间、服务器和媒体库范围。
- `manifest.json` 存在并包含工具版本、Emby 版本、schema。
- 媒体图片和人物头像数量符合源库实际情况。

### 5.2 导入预检或导入测试

1. 连接目标 Emby。
2. 选择刚导出的导出包。
3. 使用同一测试包导入到不同 Emby 版本目标。
4. 启用元数据、媒体图片、人物头像。
5. 使用明确并发配置。
6. 启动导入。
7. 观察日志是否持续输出。
8. 下载完整日志和 import report。

导入通过条件：

- 任务状态为完成。
- 未匹配和歧义数量符合预期；已知未加入媒体库的项目可以单独注明。
- 元数据成功数接近或等于应匹配项目数。
- 媒体图片成功写入，目标 Emby 页面能显示海报。
- 人物头像成功写入，目标 Emby 页面能显示人物头像。
- 失败摘要可解释，不出现大量 `source null`、`Object reference`、旧 ID 相关错误。

### 5.3 UI 与日志检查

1. 桌面宽度检查导出/导入表单布局。
2. 手机宽度检查媒体库较多时是否挤乱。
3. 日志窗口只显示最近日志，页面不卡顿。
4. 完整日志可下载。
5. 页面版本号可见。
6. 中止任务按钮可用，任务中止后后台不继续无限运行。

## 6. 必须记录的测试结果

每次远端大版本测试结束后，使用 `docs/templates/REMOTE_TEST_REPORT_TEMPLATE.md` 填写测试报告，并在开发记录或交付说明中记录：

- 测试镜像标签。
- Git commit。
- 源 Emby 版本。
- 目标 Emby 版本。
- 测试媒体库。
- 导出总结。
- 导入总结。
- 已知失败或跳过原因。
- 是否推荐用户验收。
- 如果验收通过，应更新 `docs/PROJECT_BLUEPRINT.md` 的回滚基线。

不要记录服务器地址、SSH 用户名、私钥路径、API Key 或 token。

## 7. 回滚验证

如果新版本测试失败：

1. 停止测试容器。
2. 拉取当前验收基线镜像。
3. 使用同一导出包或同一小样本重跑关键导入路径。
4. 确认可回滚版本仍能完成导入。

当前首选回滚基线见 `docs/PROJECT_BLUEPRINT.md` 的“已知回滚点”。
具体回滚命令和验证步骤见 `docs/ROLLBACK.md`。

## 8. 完成门槛

满足以下条件才可交付给用户验收：

- 本地基础检查通过。
- 远端 4.8.11 和 4.9.5 关键路径实测通过，或明确说明某版本环境不可用且已有替代证据。
- 导出/导入日志和报告无新的系统性失败。
- Docker Hub 发布状态明确。
- 回滚标签明确。
- 蓝图或关键上下文快照已更新。
