# 媒体技术信息导出/导入功能设计蓝图

## 1. 结论与边界

媒体技术信息迁移采用**无插件、同版本、两阶段离线 SQLite**方案。

- Emby `4.8.11.x` 的媒体技术信息只允许导入到 `4.8.11.x`。
- Emby `4.9.5.x` 的媒体技术信息只允许导入到 `4.9.5.x`。
- 不支持 `4.8 -> 4.9` 或 `4.9 -> 4.8` 的媒体技术信息数据库写入。
- 普通元数据、媒体图片、人物信息和人物头像继续通过 Emby API 导入，仍可跨版本迁移。
- 不依赖 MediaInfoKeeper、StrmAssistant 或其他 Emby 插件。
- 不再通过 `POST /Items/{id}` 或其他公开 Item API 回写 `MediaSources`、`MediaStreams`、`Chapters`。
- 不在 Emby 运行时直接修改 SQLite；离线写入前必须停止目标 Emby。

这里的“同版本”以兼容 profile 为准，不只比较主版本字符串。首批只开放经过实测的 `4.8.11.x -> 4.8.11.x` 与 `4.9.5.x -> 4.9.5.x`，其他补丁版本需先完成 schema 指纹和真实导入验证。

## 2. 背景与远端证据

需要迁移的媒体技术信息包括：

- `MediaSources`：容器、大小、码率、时长和媒体源摘要。
- `MediaStreams`：视频、音频、字幕流及其编码、分辨率、声道、语言等信息。
- `Chapters`：章节和片头、片尾等标记。

远端 Emby `4.8.11.0` 与 `4.9.5.0` 实测已经确认：

1. 在线向 Item 更新接口提交媒体技术字段时，HTTP 请求可以返回成功，但 Emby 回读时相关字段未持久化，属于**静默丢弃**。因此 HTTP 2xx 不能作为媒体技术信息导入成功的证据。
2. 在 Emby 停止后，按对应版本 SQLite schema 直接写入并重启回读，两版均成功恢复测试条目的 **3 条媒体流和 2 个章节**。
3. 两个大版本的内部 ID、数据库 schema 和序列化细节不同，不能把一版的 SQL 或内部标识直接套到另一版。

因此，公开 API 只用于读取、匹配、普通元数据和图片迁移；媒体技术信息的最终持久化由版本隔离的离线 SQLite adapter 完成。

## 3. 总体流程

媒体技术信息导入拆成两个明确阶段。两个阶段之间以不可变的导入计划文件交接，不能在一个在线任务中自动停服并盲写数据库。

### 3.1 阶段 A：在线匹配与计划生成

目标 Emby 保持运行，工具通过 API 完成只读工作：

1. 读取目标 Emby 版本、Server ID 和媒体库状态。
2. 校验导出包中的源 Emby 版本和 media-info schema。
3. 拒绝不受支持的跨版本媒体技术信息导入。
4. 使用现有匹配器按媒体文件、ProviderIds、剧集层级、名称和年份匹配目标 Item。
5. 回读目标 Item 的当前媒体技术信息，生成 before 摘要。
6. 为每个唯一匹配项生成离线写入计划；未匹配和歧义项不进入写集合。
7. 计算导入包、目标服务器和计划内容的校验和。
8. 输出预检报告，要求用户确认版本、目标配置挂载、写入数量和风险项。

在线阶段不修改 SQLite，也不把源 Item ID 当作目标数据库主键。

### 3.2 阶段 B：离线 SQLite 应用

离线阶段必须满足以下顺序：

1. 停止目标 Emby，并确认进程/容器已经退出。
2. 设置 `EMBY_MIGRATOR_EMBY_DB_ROOT=/emby-dbs`，将目标 Emby 的完整 `/config` 以可读写方式挂载到 `/emby-dbs/<实例名>`。
3. 再次读取数据库和配置中的版本/profile 指纹，必须与计划完全一致。
4. 备份目标数据库及同目录相关 `-wal`、`-shm` 文件；记录大小、时间和 SHA-256。
5. 以 SQLite 事务执行版本 adapter 生成的参数化写入。
6. 在提交前执行行数、外键引用和计划项数量校验；任一硬错误立即回滚事务。
7. 提交后执行 `PRAGMA quick_check` 和 `PRAGMA integrity_check`，结果必须为 `ok`。
8. 关闭所有数据库连接，写出离线应用报告。
9. 重启目标 Emby。
10. 通过 Emby API 回读每个样本及统计摘要，验证流数量、章节数量和关键字段。
11. 回读失败时停止继续操作，保留报告和备份，按恢复流程还原数据库。

工具不得在无法确认目标 Emby 已停止、目标配置挂载错误、数据库版本不匹配或备份失败时进入写事务。

## 4. 导出包设计

### 4.1 数据来源

在线导出通过 Emby API 获取 Item 详情，保留：

- `MediaSources`
- 顶层与媒体源内嵌的 `MediaStreams`
- `Chapters`
- 源 Emby 完整版本、Server ID、导出时间和 schema profile
- 用于目标匹配的路径特征、ProviderIds、类型、名称、年份和剧集层级

对于重复出现在顶层和 `MediaSources` 内的流，导出层要按稳定技术指纹去重，但保留原始来源位置供审计。

### 4.2 文件结构

当前实现沿用现有导出包结构，在 `manifest.json` 的项目记录中保存媒体技术信息，并生成独立的数据库应用计划和报告：

```text
manifest.json
media-db-plan-<target>.json
media-db-apply-<timestamp>.json
```

`manifest.json` 中每个媒体 Item 的媒体信息记录至少包含：

```json
{
  "sourceItemId": "source-only-id",
  "itemType": "Movie",
  "matchIdentity": {
    "pathFingerprint": "sha256:...",
    "providerIds": {},
    "name": "example",
    "year": 2026
  },
  "mediaInfo": {
    "mediaSources": [],
    "mediaStreams": [],
    "chapters": []
  },
  "sourceProfile": {
    "embyVersion": "4.8.11.0",
    "schemaFamily": "emby-4.8.11"
  },
  "recordHash": "sha256:..."
}
```

源 Item ID 只用于报告关联，禁止直接作为目标数据库 ID。

### 4.3 导出校验

导出报告至少记录：

- 总 Item 数、含媒体源 Item 数、含流 Item 数、含章节 Item 数。
- MediaSource、视频流、音频流、字幕流和章节总数。
- 空媒体信息数量、API 读取失败数量和重复流去重数量。
- 每条记录及整个媒体信息文件的 SHA-256。
- 源 Emby 版本和 schema family。

## 5. 在线导入计划

### 5.1 匹配规则

匹配继续复用普通迁移的稳定规则：

1. 媒体文件路径特征和文件名 stem。
2. ProviderIds、Item 类型和年份。
3. 剧集的 Series、Season、Episode 和父级关系。
4. 名称、OriginalTitle 等辅助字段。

只有唯一匹配才可生成 SQLite 写入计划。歧义、无匹配、目标 Item 已删除或目标路径不一致均标记为跳过。

### 5.2 计划文件

计划文件命名为 `media-db-plan-<target>.json`，包含：

- 计划 ID、创建时间、工具版本和计划 schema 版本。
- 导出包 SHA-256、源版本/profile。
- 目标 Emby 地址的脱敏标识、Server ID、完整版本/profile。
- 目标数据库 identity：规范路径、文件大小、mtime、schema 指纹和只读快照 hash。
- 每个源记录对应的目标 API Item ID、目标数据库内部定位信息和预期写入计数。
- 匹配依据、歧义/跳过原因、before 摘要和期望 after 摘要。
- 整份计划的 SHA-256。

离线阶段只能消费同一目标服务器、同一数据库 identity 和同一导出包生成的计划。数据库在计划生成后发生变化时必须重新预检。

## 6. 版本隔离 SQLite Adapter

### 6.1 Adapter 约束

每个 adapter 独立实现：

- schema 探测与严格版本判定。
- API Item ID 到目标数据库实体的安全解析。
- 媒体源摘要字段写入。
- MediaStreams 替换/补全策略。
- Chapters 替换/补全策略。
- 版本特有的序列化、索引和关联维护。
- 写前、写后断言与回读摘要。

首批 adapter：

- `emby-4.8.11`：只接受已验证的 `4.8.11.x` schema 指纹。
- `emby-4.9.5`：只接受已验证的 `4.9.5.x` schema 指纹。

禁止通过“字段名差不多”自动套用未知版本。未知列可以保留，但未知 schema 不允许写入。

### 6.2 写入策略

- 使用参数化 SQL，不拼接来自导出包的字符串。
- 在单个总事务或可验证的分批事务中执行；首版优先单事务保证全有或全无。
- 写入目标数据库自己的 Item 主键，不复用源端 `Id`、`MediaSourceId` 或章节内部 ID。
- 外置字幕路径必须以目标媒体路径为基准重建，禁止写入源服务器绝对路径。
- 目标存在有效信息时默认“仅补全”；覆盖模式必须在计划中明确并二次确认。
- 每个 Item 写入后立即在同一事务内查询验证数量和关键字段。
- 不执行 `VACUUM`，避免不必要的全库重写；索引维护仅按对应版本已验证行为执行。

### 6.3 数据库定位与文件范围

数据库文件必须通过相对于 `EMBY_MIGRATOR_EMBY_DB_ROOT` 的路径定位，并位于挂载的目标 Emby 配置目录内；不在代码或文档中假定单一固定文件名。备份范围至少包括：

- 实际识别出的主 SQLite 数据库。
- 同目录同名的 `-wal` 与 `-shm`（如存在）。
- 用于判断版本/schema 的必要配置文件。
- 离线导入计划和校验和文件。

备份写到 `/data/backups/<timestamp>-<server-id>/`，不能覆盖旧备份。

## 7. 安全门槛与恢复

### 7.1 写入前硬门槛

以下任一条件不满足，任务必须失败且保持数据库未修改：

- 目标 Emby 已停止且没有进程占用数据库。
- `/emby-dbs/<实例名>` 是目标实例配置目录的真实挂载，不是空目录或错误实例。
- 源、目标媒体信息 profile 相同且在白名单内。
- 当前数据库 identity 与在线计划一致。
- 数据库备份成功并通过 SHA-256 校验。
- `PRAGMA quick_check` 写前结果为 `ok`。
- 所有计划写入项均唯一定位到目标实体。

### 7.2 自动回滚

- SQL 执行、断言或提交前校验失败：回滚 SQLite 事务。
- 提交后 `integrity_check` 失败：关闭连接，将损坏副本隔离，并从本次备份恢复。
- Emby 重启失败或 API 回读严重不一致：停止 Emby，恢复备份，重新启动并再次做健康检查。
- 恢复动作和结果必须写入独立报告，不允许只留在实时日志。

### 7.3 并发与锁

离线 SQLite 应用任务全局串行。同一目标配置挂载同时只允许一个任务持有文件锁。普通 API 导入任务、扫描任务和媒体技术信息离线任务不得并行写同一目标实例。

## 8. 报告与断点

### 8.1 在线计划报告

记录：总记录、唯一匹配、未匹配、歧义、版本拒绝、目标已有信息、预计新增/替换的流和章节数量。

### 8.2 离线应用报告

记录：

- 目标版本/profile 和数据库 identity。
- 备份目录及校验结果。
- 计划数、写入数、跳过数、失败数。
- 写入前后 MediaSources、MediaStreams、Chapters 数量。
- 事务状态、`quick_check`、`integrity_check` 结果。
- Emby 重启结果和 API 回读结果。
- 恢复是否触发及最终状态。

### 8.3 断点原则

在线读取和计划生成可以断点续跑。离线数据库事务不在中间断点恢复：失败后回滚或恢复备份，再从完整计划重新执行。这样避免半套媒体流或章节处于不可解释状态。

## 9. UI 与部署设计

### 9.1 在线阶段

导入页提供“生成媒体技术信息离线计划”，展示：

- 源/目标 Emby 版本和是否同 profile。
- 唯一匹配、未匹配、歧义数量。
- 预计写入流和章节数量。
- 目标 Emby 配置目录在 `/emby-dbs/<实例名>` 下的挂载要求。

跨版本时明确显示“普通元数据和图片可继续导入；媒体技术信息不可跨版本写入”。

### 9.2 离线阶段

离线应用必须是独立危险操作，按钮文案不能与普通“开始导入”混淆。确认页展示：

- 目标实例、完整版本、计划 ID、数据库路径。
- Emby 停止状态。
- 备份位置和预计写入数量。
- “仅补全”或“覆盖”策略。

未挂载目标 Emby 配置目录、只读挂载、目标 Emby 未停止或计划失效时禁用执行按钮并给出中文原因。

### 9.3 Docker 挂载

普通 API 迁移仍只需要项目自身 `/data` 和 `/config`。启用离线媒体信息导入时，设置数据库根目录，并将目标 Emby `/config` 挂载到该根目录下的独立实例路径：

```bash
-e EMBY_MIGRATOR_EMBY_DB_ROOT=/emby-dbs \
-v /path/to/target-emby/config:/emby-dbs/target-emby
```

页面中的数据库路径必须相对于 `/emby-dbs`，例如 `target-emby/data/library.db`。不能使用绝对路径，也不能与 Emby Migrator 自身 `/config` 混用。写权限只在离线应用阶段需要；在线计划阶段按只读处理。写库前必须停止目标 Emby，并确认其进程或容器已经退出。

## 10. 测试矩阵

### 10.1 版本矩阵

| 源端 | 目标端 | 普通元数据/图片 | 媒体技术信息 |
| --- | --- | --- | --- |
| 4.8.11.x | 4.8.11.x | 支持 | 支持，使用 4.8 adapter |
| 4.9.5.x | 4.9.5.x | 支持 | 支持，使用 4.9 adapter |
| 4.8.11.x | 4.9.5.x | 支持 | 拒绝，不生成写入计划 |
| 4.9.5.x | 4.8.11.x | 支持 | 拒绝，不生成写入计划 |

### 10.2 数据用例

- 电影：3 条流、2 个章节，验证已完成的远端基线。
- 多音轨、多字幕、外置字幕、无章节、片头片尾标记。
- 剧集：同名剧集、多季多集，确认不串集。
- `.strm`：目标路径重建和媒体源摘要。
- 目标已有更完整信息：默认不覆盖。
- 无匹配和歧义：不得进入 SQL 写集合。

### 10.3 故障用例

- Emby 未停止、数据库被占用、挂载只读或挂载错实例。
- 计划生成后数据库发生变化。
- 4.8 计划用于 4.9，或反向使用。
- 备份失败、磁盘空间不足、事务中断。
- SQL 断言失败、`integrity_check` 非 `ok`。
- 重启失败、重启后 API 回读数量不一致。
- 任务重复执行，验证幂等和“仅补全”策略。

## 12. 关键决策摘要

- 媒体技术信息不依赖插件，采用停服后的 SQLite 持久化。
- 在线阶段只匹配并生成不可变计划；离线阶段才允许修改数据库。
- `4.8.11.x` 与 `4.9.5.x` 使用明确的版本门控和已验证 schema，不承诺跨版本媒体技术信息兼容。
- HTTP 成功不代表持久化成功；最终成功必须同时满足数据库完整性检查和重启 API 回读。
- 普通元数据、媒体图片、人物信息和人物头像继续走 API，并保留跨版本迁移能力。
- 备份、事务、完整性检查、重启回读和可恢复报告均为发布硬门槛，不是可选增强。
