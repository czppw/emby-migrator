# Emby Migrator Project Blueprint

本文是 Emby Migrator 后续开发的主蓝图。每一轮开发、修复、发布前，都先对照本文，确认当前动作没有偏离项目目标。

## 0. 使用规则

每轮开发前必须先做三件事：

1. 读取本文的“当前基线”“开发护栏”“路线图”。
2. 把用户本轮需求归类到 P0/P1/P2/P3，确认它是在补稳定性、性能、体验，还是长期能力。
3. 如果本轮产生新的关键事实，例如 Emby 版本兼容差异、Docker 部署约定、匹配规则变化、回滚标签、已验证问题，结束前更新“关键上下文快照”。

上下文压缩前或长任务中途，也要把新增关键信息写回本文，不能只依赖聊天记录。

每轮开发的具体执行顺序见 `docs/DEVELOPMENT_WORKFLOW.md`。

## 1. 当前基线

### 项目定位

Emby Migrator 是一个轻量 Docker Web 工具，用来在不同 Emby 服务器之间迁移媒体元数据、海报/背景图等媒体图片、人物信息、人物头像和媒体技术信息。当前产品方向明确为单用户账号与密码入口，不规划额外账号体系。

项目基线来自用户原有 Python 脚本：导出时保存完整可迁移数据，导入时不依赖旧 Emby 内部 ID，而是通过媒体文件、ProviderIds、剧集信息、名称等稳定特征匹配新 Emby 中的项目。

项目定位是按需迁移和备份恢复工具，不做常驻同步器；定时导入、双向同步不作为规划目标。

媒体技术信息（`MediaSources`、`MediaStreams`、`Chapters`、编码、码率、音轨、字幕轨等）采用无插件、同版本、两阶段离线 SQLite 方案：在线阶段通过 API 读取目标并生成不可变匹配计划，离线阶段必须停止目标 Emby、挂载其 `/config`、完成数据库备份后再以版本专用 adapter 事务写入。媒体技术信息仅支持 `4.8.11.x -> 4.8.11.x` 和 `4.9.5.x -> 4.9.5.x`；普通元数据、图片、人物与头像继续走 API，仍支持跨版本迁移。详细设计见 `docs/MEDIA_INFO_BLUEPRINT.md`。

### 当前确认目标

- 一次性完整导出/导入元数据、媒体图片、人物信息、人物头像。
- Docker 部署，默认 Web 端口 `8787`。
- 默认 host 网络模式，方便用户直接填写宿主机本地 Emby 地址或代理地址。
- 默认容器数据目录 `/data`，导出包目录 `/data/exports`。
- 推荐宿主机挂载目录 `/opt/emby-migrator/data:/data` 和 `/opt/emby-migrator/config:/config`。
- 默认登录账号 `admin`、密码 `password`，密码可用 `EMBY_MIGRATOR_PASSWORD` 修改。
- 登录页使用浏览器可识别的标准账号和密码字段；安全设置支持验证当前密码后修改单用户账号，修改成功会使旧会话失效。
- 默认时区建议 `TZ=Asia/Shanghai`。
- 当前产品方向是单用户账号与密码入口，不规划额外账号体系。
- 普通元数据、图片、人物和头像只通过 Emby API 操作；仅媒体技术信息在用户显式启用且目标 Emby 已停止时，按同版本 profile 离线修改目标 SQLite。

### 当前已具备能力

- Web 登录入口。
- Emby 地址/API Key 连接测试。
- 分别从源服务器和目标服务器读取媒体库，选择导出媒体库并在导入/预检前确认目标媒体库状态。
- 导出元数据、媒体图片、人物头像。
- 导入元数据、媒体图片、人物头像。
- 导出/导入并发选项。
- 任务日志实时展示，前端限制显示量，完整日志可下载。
- 任务中止能力。
- 服务端配置持久化：可保存服务器地址簿、当前源/目标服务器选择、默认并发和图片类型到 `/config/settings.json`，页面刷新后可复用；API 响应不回传明文 API Key。
- TG 配置入口：可保存 Bot Token、对话 ID、代理地址到 `/config/telegram.json`，支持发送中文测试消息和中文任务终态通知；导出通知显示源服务器，导入/预检通知显示源服务器和目标服务器；代理仅用于 TG 请求。
- 导出/导入总结日志，包含成功/失败统计和用时。
- 前端版本信息展示。
- 暗黑模式和移动端自适应基础优化。
- 导出包 manifest 中包含工具版本、Emby 版本、schema 信息。

### 当前兼容经验

- Emby 4.8.11 和 4.9.5 均需要保持兼容。
- 4.9.5 对元数据 payload 更严格，旧 Emby 内部 ID、旧 People/Studios 内部对象不能原样回写。
- 人物头像导入不能依赖旧 Emby 人物 ID。
- 人物头像上传应优先用 `/Persons?SearchTerm=名字` 在目标服务器查找目标人物 ID，再向 `/Items/{targetPersonId}/Images/Primary` 上传。
- 某些 4.9.5 环境中 `/Persons/{name}` 可能 HTTP 500，只能作为兼容兜底，不应作为首选路径。
- 导入时应写入干净人物字段：`Name`、`Type`、`Role`、`ProviderIds`，不能携带旧服务器人物 `Id`。

## 2. 开发护栏

### 必须保持的核心原则

- 不使用旧 Emby 内部 ID 作为跨服务器匹配依据。
- 导入匹配优先看媒体文件稳定特征，其次才是名称相似度。
- 导出和导入要一起验证，不能只做导出不测导入。
- 数据完整性优先，元数据、媒体图片、人物头像不能拆成长期残缺版本。
- 失败时日志要能解释大致原因，不能只显示“失败”。
- 大库场景要避免无限日志堆叠、长时间无进度输出、单线程慢跑。
- Docker 用户应能通过一条命令启动，默认路径和权限尽量不踩坑。
- 不把 GitHub token、Docker Hub token、Emby API Key 写入仓库或日志。

### 变更前检查清单

每次准备改代码前，先回答：

- 这次改动影响导出、导入、匹配、头像、UI、部署中的哪一块？
- 是否会改变导出包格式？如果会，是否需要更新 manifest schema 或兼容旧包？
- 是否可能影响 Emby 4.8.11 或 4.9.5？是否有测试覆盖？
- 是否会让旧服务器内部 ID 混入导入 payload？
- 是否会让大库任务更慢、日志更吵、前端更卡？
- 是否需要更新 README、Docker 命令、蓝图或关键上下文快照？

### 完成前检查清单

功能或修复完成前，至少执行：

```bash
go test ./...
node --check web/assets/app.js
go build ./cmd/server
```

如果改了前端体验，还需要用浏览器实际检查桌面和手机宽度。若改了 Docker、README 或部署命令，要确认新用户按文档启动不会遇到权限或端口说明缺失。

### 目标模式开发约定

用户采用目标模式推进本项目：用户给出目标和最终审核，开发、联调、双版本验证、日志分析、必要的子任务拆分由 Codex 主动执行。

- 每轮实现前先对照本蓝图，确认需求属于 P0/P1/P2/P3，且不破坏当前核心原则。
- 可以使用子智能体协助做代码审查、测试设计、日志分析或兼容性排查，但主线程必须汇总结论并亲自核对关键文件。
- 双版本兼容验证要覆盖 Emby 4.8.11 和 4.9.5 的关键路径，尤其是元数据 payload、媒体图片上传、人物信息和人物头像；这些 API 能力继续验证跨版本迁移。媒体技术信息另按同版本矩阵验证，只允许 4.8.11.x 同族迁移或 4.9.5.x 同族迁移。
- 涉及大版本兼容、导出包格式、匹配策略、元数据 payload、图片/头像上传的改动，必须在远端测试环境按不同 Emby 版本实际跑导出和导入验证；不能只依赖本地 mock 或单元测试作为最终结论。
- 远端大版本实测步骤以 `docs/REMOTE_VERSION_TESTING.md` 为准；每次涉及核心迁移逻辑的最终交付都要按该 runbook 记录测试镜像、Git commit、源/目标 Emby 版本、导出总结、导入总结和回滚标签。
- 用户只审核最终版本时，Codex 必须在交付前完成本地测试、必要的远端验证、发布状态确认和风险说明。
- 如果远端测试环境不可用，要明确记录阻塞原因，并尽量用本地 mock 或回归测试先覆盖可验证部分。

### 敏感连接信息规则

项目开发允许使用用户临时提供的远端服务器和密钥做双版本测试，但这些信息不得写入仓库文件、README、日志、测试快照或提交记录。

- 不在仓库中保存服务器地址、SSH 用户名、私钥路径、API Key、Docker Hub token、GitHub token。
- 不在最终回复中复述完整密钥、token 或私钥内容。
- 远端连接只用于开发验证和问题定位；测试结束后不把连接细节固化到公开文档。
- 需要记录测试结论时，只记录“远端双版本测试环境”及验证结果，不记录具体凭据。

## 3. 路线图

### P0 稳定化

这些优先级最高，目标是让项目成为“可靠迁移工具”，而不是临时脚本页面。

- 配置持久化：已支持保存服务器地址簿、当前源/目标服务器选择、默认并发数和图片类型到服务端 `/config`；导出路径偏好和更细配置管理仍待补充。
- 更完整任务控制：已支持中止、服务端串行任务队列、任务列表、暂停和继续；暂停是协作式暂停，已发出的 Emby 请求会先完成，新的项目/头像任务不再继续派发。
- 导入预检：先跑匹配预览，展示未匹配、歧义、预计写入数量，再允许正式导入。
- Emby 版本兼容 profile：识别源/目标 Emby 版本，集中管理 4.8/4.9 的 payload 和人物接口差异。
- 导出包校验：已校验导出包路径安全、文件存在性、文件大小和 SHA256；后续可补包级文件数量、图片数量和 schema 迁移兼容策略。
- 回归测试矩阵：固定 4.8.11、4.9.5 的关键 API 行为 mock，防止修新版破坏旧版。
- 媒体技术信息离线迁移：按 `docs/MEDIA_INFO_BLUEPRINT.md` 完成在线匹配计划、4.8/4.9 隔离 SQLite adapter、停服与挂载校验、数据库备份、事务写入、完整性检查、重启回读和失败恢复；禁止恢复已证实无效的 Item API 回写路径。

### P1 性能和可靠性

目标是大媒体库可控、可恢复、速度合理。

- 分通道并发：元数据、媒体图片、人物头像分别设置并发。
- 增量导出/导入：只处理新增或变更项目，减少重复全量迁移。
- 缓存目标服务器查询：人物、ProviderIds、媒体路径搜索结果避免重复请求。
- 重试与限流策略：针对 HTTP 429/500/502/503/504、超时、慢请求做统一退避。
- 任务断点：导入中断后能从报告或状态文件继续。当前尚未完成完整断点续跑。
- 失败报告结构化：失败摘要保留少量示例，完整失败列表写入 report 文件。

### P2 用户体验

目标是让普通 Docker 用户不用理解内部细节也能安全完成迁移。

- 服务器地址簿：已完成基础可用版。可保存多个 Emby 服务器条目，并按 source / target 角色切换当前来源和目标；更细的批量管理与校验提示仍待增强。
- 导入后对比报告：导入前后项目数、图片数、人物头像数、缺失项对比。
- 更清晰的任务页面：已支持任务列表和当前任务切换；运行中、已完成、失败、已取消的完整分组视图仍待补。
- UI 表单压缩与响应式优化：媒体库多、导入包多、移动端窄屏都不能挤乱。
- 图片类型中文化：用清晰中文名称展示 Primary/Backdrop/Logo/Thumb 等，并默认全选。
- 新手部署提示：权限、host 网络、端口占用、数据目录位置给出明确诊断。

### P3 长期能力

这些属于成熟工具方向，不应挤占 P0/P1。

- 定时备份。
- TG 通知：已支持保存配置、中文测试消息、任务 done/failed/stopped 中文终态通知；通知包含对应服务器地址字段；代理仅用于 TG 请求，API 不返回明文 Bot Token。
- 匹配规则模板：按媒体文件、ProviderIds、名称、OriginalTitle、年份、季集信息组合。
- 只读审计模式：只分析源/目标差异，不写入目标 Emby。

不规划普通导入内容撤销。媒体技术信息离线写入必须具备数据库备份恢复能力，但不承诺跨版本转换；导入前必须通过在线预检、版本/profile 校验、计划校验和数据库备份降低风险。

### 当前增强项完成度快照

截至本轮本地开发状态：

- 暂停/继续任务、任务队列：已完成基础可用版。服务端串行队列、任务列表、暂停、继续、中止已接入；暂停为协作式，不会硬中断已发出的 Emby HTTP 请求。
- 增量导出 / 导入增量包：基础路径已完成并通过远端双版本专项验证。导出侧可基于最近基线跳过未变化项目，并在 `manifest.json` 记录 incremental 元数据；导入侧会跳过增量包中标记未变化的项目。独立的按目标库变更集执行增量导入尚未完成。
- 更完整的断点续跑：基础 resume 路径已完成并通过远端专项验证。当前 `resume` 会读取最近一次 `import-report-*.json`，跳过已成功项目和头像；但没有完整状态文件，也不能精确恢复进行中的元数据、媒体图片或人物头像步骤。
- 服务器地址簿：已完成基础可用版。已支持保存多个 Emby 服务器条目、按 source / target 角色选择并切换，并可分别从源/目标服务器读取媒体库；更细的批量管理与导出路径偏好仍待增强。
- 导入后差异对比报告：未完成。当前只有导入/预检报告和摘要，没有 before/after 差异快照。
- 更结构化的失败报告：部分完成。报告内已有兼容信息、before/after 摘要、匹配/未匹配/歧义/错误统计、失败示例，以及媒体图片和人物头像失败统计；完整失败分组、风险等级、完整失败清单仍待增强。
- TG 通知：已完成基础可用版。支持配置保存、中文测试消息、中文任务终态通知、服务器地址字段和代理仅用于 TG 请求。
- 单用户账号与密码入口：已完成并固定为当前产品方向，不规划额外账号体系。
- 导入内容撤销：不规划。当前仅保留 Docker 镜像/版本回滚文档，不做导入内容撤销。
- 媒体技术信息：方案已从在线 Item API 回写调整为无插件、同版本、两阶段离线 SQLite。远端已验证在线 HTTP 会静默丢弃字段，直接数据库写入在 4.8.11.0 和 4.9.5.0 均可恢复 3 条流、2 个章节；版本专用 adapter、停服编排、备份恢复和完整发布级回归仍需按专项蓝图完成。

## 4. 架构方向

### 后端

- Go 继续作为主语言，保持单二进制、轻量 Docker 镜像。
- 业务边界保持清晰：`internal/emby` 只负责 Emby API；`internal/exporter` 负责导出/导入流程；`internal/storage` 负责包结构和 manifest；`internal/web` 负责 HTTP/API/UI。
- 新增兼容逻辑优先集中到 Emby client 或 exporter 的专门 helper，不要在多个地方复制版本判断。
- 长任务都必须支持 context cancel，不能让页面中止后后台继续无限跑。
- 媒体技术信息离线写入应使用独立 SQLite adapter 边界；4.8.11 与 4.9.5 的 schema 探测、SQL、断言和回读逻辑互不复用，未知 schema 默认拒绝写入。
- 在线匹配计划与离线应用任务分离；离线任务必须验证停服、目标 `/config`、计划/database identity、备份和文件锁。

### 前端

- 不做营销页，首页就是可用工具。
- 表单以效率为主，导出/导入并排或紧凑布局，适配移动端。
- 日志窗口只显示最近部分，完整日志用下载。
- 所有影响写入的危险操作要明确显示目标服务器和导入包。
- 版本号始终可见，便于用户反馈问题时定位镜像版本。

### 导出包

导出包必须长期可读、可迁移、可诊断。核心内容：

- `manifest.json`：包级索引、工具版本、Emby 版本、schema、统计信息。
- 每个项目的 `info.json`：清洗后的结构化元数据。
- 每个项目的 `raw.json`：必要时保留源信息，但导入不能盲目原样回写。
- 媒体图片文件。
- 人物头像文件。
- 导入报告：每次导入生成独立 report，记录匹配结果和失败摘要。

`raw.json` 可能包含 Emby 返回的媒体技术信息，但不再作为 Item API payload 原样回写。媒体技术信息导出应生成带版本 profile、记录校验和和稳定匹配特征的独立快照；在线导入只生成匹配计划，最终由同版本离线 SQLite adapter 消费计划。

## 5. 匹配策略

### 媒体项目

匹配顺序应保持稳定，并尽量可解释：

1. 媒体文件路径/文件名 stem。
2. ProviderIds 重合。
3. 剧集的 SeriesName、Season、Episode、父级路径信息。
4. 名称、OriginalTitle、年份等辅助匹配。
5. 多候选无法确认时标记歧义，不应随便写入。

导入和媒体库名字不应强绑定。新旧 Emby 的媒体库名称、内部库 ID 都可能变化，核心判断应围绕实际媒体项目。

### 人物头像

人物头像不能用旧服务器人物 ID 导入。当前策略：

1. 导入媒体项目元数据时写入干净 People，让目标 Emby 生成/关联人物。
2. 头像阶段按人物名字和 ProviderIds 在目标 Emby 查找人物。
3. 找到目标人物 ID 后上传 Primary 头像。
4. 查不到人物时记录失败，不阻塞其他头像。

## 6. 发布和回滚

### Docker 发布

主发布方式是 Docker Hub 镜像：

```text
czppwa/emby-migrator:latest
czppwa/emby-migrator:sha-xxxxxxx
```

GitHub main 推送后由 Actions 构建并推送 Docker Hub。发布后要确认：

- GitHub Actions 成功。
- Docker Hub 出现对应 `sha-xxxxxxx` 标签。
- `latest` 更新时间符合本次发布。

### 推荐运行命令

```bash
docker run -d \
  --name emby-migrator \
  --restart unless-stopped \
  --network host \
  -e TZ=Asia/Shanghai \
  -e EMBY_MIGRATOR_PASSWORD=password \
  -v /opt/emby-migrator/data:/data \
  -v /opt/emby-migrator/config:/config \
  czppwa/emby-migrator:latest
```

## 7. 关键上下文快照

本节用于防止聊天上下文压缩后丢失重要信息。只记录会影响后续开发判断的事实。

- 用户目标：做成 Docker Web 项目，用户通过页面填写 Emby 地址、端口、API Key，自主选择媒体库导出/导入。
- 用户非常重视一次性完整性：元数据、媒体图片、人物头像要一起做，便于一次性测试。
- 用户要求导入不能依赖 Emby 内部 ID，因为不同服务器内部 ID 会变化。
- 用户原脚本的重要思路：按名称、媒体文件和稳定信息匹配新服务器项目，而不是按旧 ID。
- 用户希望日志实时可见，但前端不能无限堆叠；完整日志应可下载。
- 用户希望日志带北京时间，结尾有简要总结和用时。
- 用户希望支持中止任务，并发可配置，不限制死并发上限。
- 用户希望 Docker 使用 host 网络，默认 `/opt/emby-migrator/data` 和 `/opt/emby-migrator/config` 映射到容器 `/data`、`/config`。
- 用户希望页面显示版本信息。
- 用户希望 Docker 用户刷新页面后不用重新输入 API Key；当前已增加服务端 `/config/settings.json` 保存服务器地址簿、源/目标服务器选择和默认任务选项，响应只返回 API Key 掩码。
- TG 通知规划只保留 TG；当前已增加 TG 配置、中文测试消息、中文任务终态通知和服务器地址字段，代理地址仅用于访问 TG，不影响 Emby 连接。
- 不规划导入内容撤销；项目只保留 Docker 镜像/版本回滚，不实现导入内容撤销。
- 用户采用目标模式：后续整体开发、测试、双版本验证由 Codex 主动执行，用户主要审核最终版本。
- 当前已记录的可回滚验收基线为 `0.1.0-beta.3` / Docker Hub `czppwa/emby-migrator:sha-97ba9d8`，对应 Git commit `97ba9d88cd6096c7b67cf6096652da92b567a362`；后续开发若出问题，优先回滚到该标签，必要时再退到上一稳定点 `sha-7dcc0e1`。
- 测试大版本兼容时必须使用远端测试环境按不同 Emby 版本做实际导出/导入验证；本地 mock 只能作为提前防线，不能替代最终大版本实测。
- 用户提供过临时远端服务器和 SSH 私钥用于双版本开发测试；该类连接信息只能作为本地/会话敏感信息使用，不得写进仓库或公开文档。
- 远端双版本测试环境当前重点媒体库为 `日韩剧集` 和 `日韩电影`；后续新增测试媒体库沿用同一媒体地址/路径约定，测试匹配逻辑时不能依赖媒体库内部 ID 或媒体库名称强绑定。
- 远端测试环境提供 mihomo 代理端口 `7890`；后续需要连接 TG 或外网资源时，可以在远端测试流程中使用该代理。
- 2026-06-18 远端隔离实例已完成 4.8.11/4.9.5 全量导出导入、跨版本导入、TG 中文通知、增量导出/导入增量包和 resume 续跑基础路径验证；`0.1.0-beta.3` 已发布到 Docker Hub，发布前报告见 `docs/PRE_RELEASE_TEST_REPORT_20260618.md`。
- 2026-07-14 媒体技术信息远端实测确认：Emby 4.8.11.0 和 4.9.5.0 的在线 Item 更新请求会静默丢弃 `MediaStreams`/`Chapters`；停服后使用对应版本 schema 直接写入 SQLite，两版重启回读均成功恢复 3 条媒体流和 2 个章节。因此媒体技术信息改为无插件、同版本、两阶段离线 SQLite，不再承诺跨版本；普通元数据和图片的 API 跨版本能力不受影响。
- 用户反馈过 `/data/exports permission denied`，因此文档和部署逻辑要持续关注挂载目录权限。
- Emby 4.9.5 曾出现 `/Persons/{name}` HTTP 500，头像上传已改为优先 `/Persons?SearchTerm=`。
- 4.9.5 元数据回写曾出现 `source null`/HTTP 400，payload 需要清洗，不能原样写旧 raw。
- 不要在仓库、README、日志中保存或复述 Docker Hub token、GitHub token、Emby API Key。
